# ADR-006 — PlanValidator: Validation Layer Between Planner and Executor

**Status:** Accepted  
**Date:** 2026-06-25  
**Authors:** Architecture team

---

## Context

The execution flow for a Mission is:

```
Intent → Mission → Planner → [plan steps] → Executor → tools
```

The Planner generates a sequence of steps. Each step references a tool name and provides parameters. The Executor calls the tool.

In the Phase 2A implementation, plan steps generated by the LLM go directly to the Executor without validation. If the LLM generates a step referencing a non-existent tool, or a step that requires permissions the workspace does not hold, the error is discovered at execution time — when the Executor tries to call the tool and fails.

---

## Problem

Execution-time failures for planning errors are worse than validation-time failures for several reasons:

1. **Wasted execution.** A mission may execute 5 steps successfully before reaching a step that fails because the tool does not exist. The 5 successful steps may have had side effects (files written, API calls made, notifications sent) that cannot be easily undone.

2. **User confusion.** A mission that fails on step 6 of 10 because of a tool name typo in the plan is difficult to explain. "The plan was invalid" is a clearer failure reason than "step 6 returned an error."

3. **Permission escalation.** Without validation, a plan could include a tool that exists but that the workspace's permission set does not authorize. This is a security boundary, not just a runtime error.

4. **No approval safety.** When a mission requires human approval (`WAITING_APPROVAL` state), the human is reviewing a plan. If the plan is invalid, the human approves an unexecutable plan. The validation must happen before approval is requested.

5. **LLM hallucination pathway.** LLMs hallucinate tool names. The LLM Planner may generate `docker.container_restart` instead of the registered `docker.restart`. Without a validation step, this error propagates silently until execution.

---

## Why This Generates Future Difficulty

As the number of registered tools grows (10 → 50 → 200+), the probability that a generated plan contains an invalid tool reference increases. LLM-generated plans will have a non-zero hallucination rate for tool names regardless of prompt quality. The validation layer is necessary to keep execution reliable as the capability surface grows.

Furthermore, the planned `WorkflowPlanProvider` generates deterministic plans from templates. These templates could contain invalid tool references if the workflow was authored against a different version of a plugin. The same validator that catches LLM hallucinations also catches template drift.

---

## Alternatives Considered

### Alternative A: Validate inside the Executor at call time
Each tool call in the Executor checks if the tool exists and if permissions are satisfied before calling.

**Advantages:** Simple. One less component.

**Disadvantages:**
- Partial execution with side effects (as described above).
- Errors discovered late, not early.
- No meaningful "plan is invalid" message — only "step 3 failed."
- Cannot be used to block approval flow before human review.

### Alternative B: Validate inside the PlanProvider
Each PlanProvider validates its own output before returning.

**Advantages:** Validation happens close to generation.

**Disadvantages:**
- Each PlanProvider must re-implement the same validation logic.
- `WorkflowPlanProvider` and `ManualPlanProvider` still need validation — they do not have LLM context about what tools exist.
- PlanProviders should not need to depend on the CapabilityRegistry; that creates a circular dependency where the provider needs to know the full execution environment.

### Alternative C: Dedicated PlanValidator between Planner and Executor (chosen)
A dedicated `PlanValidator` component receives the output of any `PlanProvider` and validates it before the plan is stored or submitted for approval.

**Advantages:**
- Single validation implementation, shared by all PlanProviders.
- Validation happens before approval, before any side effects.
- Clear failure mode: mission transitions to `FAILED` with `reason = plan_invalid` before entering `READY` or `WAITING_APPROVAL`.
- Validation result is auditable (stored as validation errors on the `ExecutionPlan`).
- New validation rules (budget limits, step count limits, forbidden tool blacklist) can be added to one place.

**Disadvantages:**
- Additional component to maintain.
- All PlanProviders must pass through the validator.

---

## Decision

Adopt **Alternative C**.

The `PlanValidator` is a synchronous component called by the `MissionEngine` after any `PlanProvider` returns a plan, and before any state transition to `WAITING_APPROVAL` or `READY`.

---

## Validation Contract

```python
@dataclass
class ValidationError:
    step_index: int | None     # None = plan-level error
    code: str                  # "tool_not_found" | "permission_denied" | "missing_parameter" | ...
    message: str
    detail: dict               # machine-readable context

@dataclass
class ValidationResult:
    valid: bool
    errors: list[ValidationError]
    warnings: list[ValidationError]   # non-fatal, plan proceeds but logged
```

```python
class PlanValidator:
    def validate(
        self,
        plan: list[MissionStep],
        mission: Mission,
        registry: CapabilityRegistry,
        workspace_permissions: set[str],
    ) -> ValidationResult: ...
```

---

## Validation Rules

| Rule | Error Code | Fatal |
|---|---|---|
| Tool name does not exist in registry | `tool_not_found` | Yes |
| Tool's capability requires a permission not held by workspace | `permission_denied` | Yes |
| Required parameter is missing or null | `missing_required_parameter` | Yes |
| Parameter type mismatch | `parameter_type_error` | Yes |
| Tool's `required_context` is not available | `missing_context` | Yes |
| Dependent capability not registered | `dependency_not_registered` | Yes |
| Plan has zero steps | `empty_plan` | Yes |
| Plan exceeds maximum step count (configurable) | `plan_too_long` | No (warning) |
| Step references an optional tool with low availability | `tool_availability_warning` | No (warning) |

---

## Position in the Mission Lifecycle

```
PlanProvider.create_execution_plan(mission)
    ↓
PlanValidator.validate(plan, mission, registry, permissions)
    ├── ValidationResult.valid = True
    │       ↓
    │   MissionEngine transitions to WAITING_APPROVAL (if requires_approval)
    │   or READY (if auto-approve)
    │
    └── ValidationResult.valid = False
            ↓
        MissionEngine transitions to FAILED
        validation_errors stored on ExecutionPlan
        event published: mission.plan_validation_failed
```

---

## Integration with ExecutionPlan Entity

The `ExecutionPlan` entity (introduced in ADR-001) stores the validation result:

```python
class ExecutionPlan(Base):
    id: UUID
    mission_id: UUID
    version: int
    status: str       # "draft" | "validated" | "approved" | "running" | "completed" | "failed" | "superseded"
    validation_errors: list[dict] | None   # JSONB; populated by PlanValidator
    created_at: datetime
    approved_at: datetime | None
    approved_by: str | None
```

A plan with `validation_errors` that are all warnings (no fatal errors) has `status = "validated"`. A plan with any fatal error has `status = "failed"` and never transitions further.

---

## Consequences

**Positive:**
- All planning errors are caught before any execution side effect.
- Human approval is only requested on valid plans.
- LLM hallucinated tool names are caught at validation, not at execution.
- Validation result is auditable — "why did this plan fail?" has a stored answer.
- Security boundary: permission checks happen in one place.
- New validation rules can be added without touching PlanProviders or the Executor.

**Negative:**
- All PlanProviders must route through the validator (cannot bypass).
- Validation adds latency to the planning phase (negligible for local checks against an in-memory registry).
- The `required_context` validation requires the validator to know what context is available, which means `MissionContext` must be passed to validation.
