# ADR-007 — StepExecutor, ExecutionContext e Transaction Policies

**Status:** Accepted  
**Date:** 2026-06-26  
**Deciders:** Victor Mendes  

---

## Contexto

O `MissionEngine.run()` original executava steps de forma primitiva:
- `tool.fn(**step.input)` chamado diretamente — sem resolução de variáveis entre steps
- Falha sempre resultava em retry hardcoded (sem configuração por step)
- Não havia objeto de estado acumulado durante a execução — outputs de step N não chegavam ao step N+1
- `eval()` aberto no `SchedulerEngine._evaluate_rule()` era vulnerável a RCE via `__subclasses__()`

## Decisão

### ExecutionContext

Introduzido `engines/execution/context.py`:

- Objeto **em memória** criado no início de cada `run()` e descartado no fim
- Acumula outputs via `variables["step_N"]` — step N+1 acessa `{{ step_0.url }}` no seu `input`
- Método `as_resolve_context()` retorna dict flat usado pela resolução de `{{ }}`
- **Distinto** de `MissionContext` (entidade de banco que guarda configuração inicial)

### StepExecutor

Introduzido `engines/execution/executor.py`:

- Responsabilidade única: executar **um** step dentro de um `ExecutionContext`
- Resolve `{{ chave.aninhada }}` no input antes de chamar a tool
- Atualiza status do step no banco (RUNNING → SUCCEEDED | FAILED)
- Publica STEP_STARTED / STEP_COMPLETED / STEP_FAILED via `PublishFn` injetada
- **Não decide** retry/abort/skip — retorna `StepResult` e deixa o `MissionEngine` decidir

### FailurePolicy (Transaction Policies por step)

Adicionado campo `failure_policy` em `MissionStep` (migração 012):

| Valor | Comportamento |
|-------|--------------|
| `retry` | Incrementa `retry_count`, reexecuta uma vez (default) |
| `abort` | Falha a missão imediatamente |
| `skip` | Marca step como SKIPPED, continua com `{}` no contexto |
| `ignore` | Igual a `skip` — registra no log mas não interrompe |

`_MAX_RETRIES = 3` definido em `engines/mission/engine.py`.

### Correção de eval() inseguro

`SchedulerEngine._evaluate_rule()` substituído por `_safe_eval_rule()`:

1. Parseia a expressão com `ast.parse(expr, mode="eval")`
2. Caminha todos os nós com `ast.walk()` e rejeita qualquer tipo fora de `_SAFE_AST_NODES`
3. Whitelist permite: constantes, nomes, comparações, operadores booleanos/aritméticos, `if/else` ternário, tuplas e listas literais
4. Proíbe explicitamente: `Call`, `Attribute`, `Subscript`, `Lambda`, comprehensions, `Import`
5. Mesmo após validação, passa `{"__builtins__": {}}` ao `eval()` por defesa em profundidade

## Alternativas consideradas

**Context via banco a cada step:** acesso mais lento, acoplamento desnecessário para estado efêmero de execução. Rejeitado.

**FailurePolicy global por missão:** granularidade insuficiente — steps diferentes têm tolerâncias diferentes. Rejeitado.

**Trocar eval() por parser próprio completo:** custo alto para o benefício. A validação de AST com whitelist é segura o suficiente para expressões booleanas simples. Aceito para revisão futura se as expressões ficarem mais complexas.

## Consequências

- Outputs de steps fluem automaticamente para steps seguintes via `{{ }}`
- Comportamento de falha é declarativo e auditável por step
- `MissionEngine` orquestra; `StepExecutor` executa — separação clara de responsabilidades
- `SchedulerEngine` não expõe mais vetor de RCE via eval aberto
- Migração 012 é não-destrutiva (coluna com `server_default="retry"`)
