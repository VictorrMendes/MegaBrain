# ADR-009 — Cognitive Orchestrator

**Status:** Accepted  
**Date:** 2026-06-27  
**Phase:** 8.5 — Cognitive Core

---

## Context

Khonshu phases 1–8B established a set of independent, capable engines:
ContextBuilder, MemoryEngine, KnowledgeEngine, SearchEngine, MissionEngine,
ExecutionEngine, CapabilityRegistry, and CognitiveLoop. Each engine works
correctly in isolation. However, every user request entering via the
conversation router followed the same fixed pipeline:

```
User → ContextBuilder → LLM → Workers (async, event-driven)
```

There was no component that decided *which* engines to activate per
request, or that generated a structured trace of the reasoning process.
The system was capable but not adaptive.

---

## Decision

Introduce a **Cognitive Orchestrator** layer in `kernel/orchestrator/` with
three new components:

### DecisionEngine
Uses a fast, deterministic LLM call (`ExecutionProfile.ROUTING`, max 5s) to
analyze each user message and return a typed `Decision` specifying which
engines should participate. This replaces implicit, hard-coded engine calls
with explicit, observable routing.

### LearningEngine
After every exchange, analyzes the conversation and decides what is worth
persisting. Returns typed `LearningAction` items that the orchestrator
applies via existing MemoryEngine and KnowledgeEngine. Keeps learning
decoupled from execution.

### CognitiveOrchestrator
Coordinates the full pipeline in a single `execute()` call:

```
OrchestratorRequest
  → ContextBuilder (always)
  → DecisionEngine (always)
  → SearchEngine (if need_search)
  → MissionEngine (if need_mission)
  → LLMProvider (always)
  → LearningEngine (if need_learning)
  → OrchestratorResponse
```

Every step is recorded as a `TraceNode` in a `ReasoningTrace`. The full
trace is returned to the API consumer, enabling the frontend to show exactly
how Khonshu reasoned before responding.

---

## Architecture Principles

**No engine knows about the orchestrator.** Dependency direction is strictly
one-way: `router → orchestrator → engines`. This preserves the independence
of each engine and avoids circular imports.

**The orchestrator does not replace existing endpoints.** The streaming
`/messages/stream` endpoint remains the primary interactive interface.
`/orchestrator/{workspace_id}/execute` is a parallel endpoint for flows
where reasoning transparency matters more than streaming speed.

**No new business logic.** The orchestrator coordinates; it never executes.
Persistence is always delegated to the appropriate engine.

**Single-user design.** No authentication, no RBAC, no workspace isolation
logic in the orchestrator itself. The `workspace_id` is passed through to
each engine call, consistent with how the rest of the system operates.

---

## Alternatives Considered

### Hard-code routing per message type
Rejected. Pattern matching on message content is brittle and doesn't scale.
LLM-based routing is more flexible and self-documenting.

### Add orchestration logic to the conversation router
Rejected. The router handles HTTP concerns; adding decision logic there
would violate Single Responsibility and make the router impossible to test
in isolation.

### Replace streaming endpoint with orchestrator
Rejected. Streaming is essential for interactive UX. Non-streaming
orchestration serves a different use case (transparency, automation).

---

## Integration Points

| Component | How it connects |
|-----------|-----------------|
| ContextBuilder | Called first, provides base system prompt |
| DecisionEngine | Called with user message, returns routing Decision |
| SearchEngine | Called only when `decision.need_search` is true |
| MissionEngine | Called only when `decision.need_mission` is true |
| LLMProvider | Called with enriched prompt, returns GenerateResult |
| LearningEngine | Called after response, produces LearningDecision |
| MemoryEngine | Used by orchestrator to apply learning actions |
| KnowledgeEngine | Used by orchestrator to apply learning actions |
| CognitiveMetrics | Updated at each pipeline step |

---

## Consequences

**Positive:**
- Every request now has a complete, structured reasoning trace
- Routing decisions are explicit and observable
- Learning is decoupled from conversation flow
- Frontend can display "how Khonshu thought" for any request
- SearchEngine and MissionEngine are now activated adaptively, not by convention

**Negative:**
- Each orchestrator call requires 2 additional LLM round-trips (decision + learning)
- Decision LLM call adds ~1–5s latency vs. the streaming endpoint
- The non-streaming endpoint is not suitable for interactive chat

**Mitigations:**
- `ExecutionProfile.ROUTING` uses a fast, small model configuration
- Decision and Learning profiles use `deterministic=True` for consistency
- The streaming endpoint remains unchanged — latency impact is zero for
  existing interactive flows
