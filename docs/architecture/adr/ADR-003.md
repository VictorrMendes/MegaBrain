# ADR-003 — Provider Abstraction Layer

**Status:** Accepted  
**Date:** 2026-06-25  
**Authors:** Architecture team

---

## Context

PAIOS is explicitly designed to survive technology replacement. The LLM (currently Ollama/qwen3), the event bus (currently PostgreSQL LISTEN/NOTIFY), the storage backend, and the notification system are all expected to change over the system's lifetime.

Without a formal abstraction layer, each domain engine that uses Ollama directly will need to be updated when Ollama is replaced. Multiplied across all engines, this becomes a significant change surface.

---

## Problem

Currently, `OllamaProvider` is imported directly in `core/dependencies.py` and injected into engines. If Ollama is replaced by a different LLM server (e.g., llama.cpp HTTP server, vLLM, or a cloud API for testing), the change propagates to every engine that was typed against `OllamaProvider`.

The problem is not specific to LLM:
- If PostgreSQL LISTEN/NOTIFY is replaced by Redis Streams, the EventBus implementation changes — but every component that imported `EventBus` directly would be affected.
- If local filesystem storage for artifacts is replaced by S3, every component that wrote files would need updating.

This violates the dependency inversion principle: domain logic should depend on abstractions, not on concrete implementations.

---

## Why This Generates Future Difficulty

As PAIOS grows, the number of provider touch-points increases. Adding a second LLM for plan generation while using the first for chat, or routing different workspaces to different models, requires that every engine accept a `LLMProvider` interface — not a specific `OllamaProvider` class.

Without the abstraction:
- No testability (cannot mock the LLM for unit tests without monkey-patching).
- No multi-provider routing.
- Every technology change cascades through the codebase.

---

## Alternatives Considered

### Alternative A: Concrete imports everywhere
All engines import `OllamaProvider` directly. Technology changes are handled by find-and-replace.

**Advantages:** Zero overhead. Simple.

**Disadvantages:** Brittle. Untestable. Cascading changes on every technology swap.

### Alternative B: Dependency injection without formal interfaces
Engines receive their dependencies via constructor injection, typed as `Any` or using duck typing.

**Advantages:** Flexible. No interface boilerplate.

**Disadvantages:** No compile-time or lint-time verification that a replacement implements the required contract. Errors appear at runtime.

### Alternative C: Python Protocol interfaces (chosen)
Each provider category is defined as a `Protocol` class. Engines depend exclusively on the protocol. Concrete implementations (OllamaProvider, LocalStorageProvider) are wired in `core/dependencies.py`.

**Advantages:**
- Structural typing: any class that implements the required methods satisfies the protocol, without inheritance.
- Replaceable at the injection point only.
- Testable: mock implementations trivially satisfy the protocol.
- IDE type checking catches missing methods.

**Disadvantages:**
- Protocol boilerplate for each provider type.
- Runtime `isinstance()` checks against Protocols require `runtime_checkable` decorator.

---

## Decision

Adopt **Alternative C**. All providers are defined as `Protocol` classes in `kernel/providers/base.py` and `kernel/events/bus.py`. Concrete implementations are only referenced in `core/dependencies.py`. No engine imports a concrete provider class.

---

## Defined Provider Protocols

```python
class LLMProvider(Protocol):
    async def generate(prompt: str, system: str | None) -> GenerateResult: ...
    async def stream(prompt: str, system: str | None) -> AsyncIterator[str]: ...
    async def chat(messages: list[ChatMessage]) -> GenerateResult: ...
    async def chat_stream(messages: list[ChatMessage]) -> AsyncIterator[str]: ...

class EmbeddingProvider(Protocol):
    async def embed(text: str) -> EmbedResult: ...
    async def embed_batch(texts: list[str]) -> list[EmbedResult]: ...

class EventBusProvider(Protocol):
    async def publish_event(event: KhonshuEvent) -> None: ...
    def subscribe_event(event_type: str, handler: TypedEventHandler) -> None: ...
    async def connect() -> None: ...
    async def disconnect() -> None: ...

class StorageProvider(Protocol):
    async def store(key: str, data: bytes, metadata: dict) -> str: ...
    async def retrieve(uri: str) -> bytes: ...
    async def delete(uri: str) -> None: ...

class NotificationProvider(Protocol):
    async def notify(title: str, body: str, priority: int) -> None: ...

class SearchProvider(Protocol):
    async def search(query: str, limit: int) -> list[SearchResult]: ...

class PlannerProvider(Protocol):
    name: str
    async def create_execution_plan(mission: Mission) -> ExecutionPlan: ...

class MetricsProvider(Protocol):
    def counter(name: str, value: int, tags: dict) -> None: ...
    def histogram(name: str, value: float, tags: dict) -> None: ...
    def gauge(name: str, value: float, tags: dict) -> None: ...
```

---

## Consequences

**Positive:**
- Any provider can be replaced by changing one function in `core/dependencies.py`.
- Multi-provider routing (different LLMs per workspace) is implementable without touching engines.
- Unit tests can use simple mock implementations.
- The system can run in a no-LLM mode (stub provider) for infrastructure testing.

**Negative:**
- Each new provider requires a Protocol definition.
- Protocol methods must be kept in sync with concrete implementations (no automated enforcement beyond type checking).

---

## Impact on Other Modules

| Module | Impact |
|---|---|
| `kernel/providers/base.py` | Existing `LLMProvider` and `EmbeddingProvider` protocols — no change |
| `kernel/events/bus.py` | `EventBus` should satisfy `EventBusProvider` Protocol |
| `core/dependencies.py` | All provider construction and injection centralized here |
| All engines | Must not import concrete provider classes |
| `StorageProvider` | Not yet implemented — required for Inbox and Artifacts |
| `MetricsProvider` | Not yet implemented — required for Observability |
