# ADR-005 — Knowledge Engine: Observations as Distinct from Facts

**Status:** Accepted  
**Date:** 2026-06-25  
**Authors:** Architecture team

---

## Context

The Knowledge Engine stores information the system learns about the world: entities (people, places, services, concepts), relations between them, and statements of varying reliability. The original model used a single `Fact` entity for all statements.

As the system processes conversations, documents, missions, and external events, it generates two categories of knowledge that have fundamentally different reliability characteristics:

1. **Verified information** — retrieved from a primary source, stated explicitly, confirmed by the user. "The PostgreSQL instance is running on port 5432."

2. **Inferred patterns** — derived from observation of behavior, statistics, or heuristics. "This user tends to prefer morning summaries." "Container restarts correlate with high memory usage after 48h uptime."

Treating both as `Fact` collapses this distinction, leading to over-confidence in inferred knowledge.

---

## Problem

When the system presents information derived from pattern inference with the same confidence as information stated by the user, two problems arise:

1. **False precision.** "Your most productive hours are 10–12h" stated as a fact may be wrong. Stated as an observation with 0.6 confidence and a 30-day expiry, it is an appropriately hedged estimate.

2. **Stale inference persists indefinitely.** A fact has no expiry — it represents something true unless explicitly corrected. An observation derived from 30 days of data should expire when the underlying pattern changes. Without a separate type, observations either live forever (wrong) or must be manually curated (impractical).

3. **Prompt injection risk.** If the Prompt Engine injects all facts into the LLM context equally, a low-confidence inference is presented with the same authority as a verified fact. The LLM will respond as if both are equally reliable.

4. **Cannot audit inference quality.** There is no way to ask "which of our stored knowledge is inferred vs. verified?" or "how accurate are our observations over time?" without a type distinction.

5. **No source traceability.** Facts should carry a source (conversation ID, document ID, user statement). Observations should carry the derivation method (which agent produced it, from which data). A single entity cannot represent both.

---

## Why This Generates Future Difficulty

The Knowledge Engine is one of the primary inputs to the Prompt Engine. As PAIOS processes more data, the proportion of inferred knowledge grows faster than the proportion of directly-stated knowledge. Without the Observation type:

- The Prompt Engine cannot filter by reliability when building context.
- The system cannot implement confidence decay (reducing confidence in stale observations).
- A "knowledge audit" feature becomes impossible to implement correctly.
- The transition to a more sophisticated inference model (Bayesian updates, source credibility weighting) requires a schema migration that introduces the concepts this ADR defines now.

---

## Alternatives Considered

### Alternative A: Single Fact entity with a `confidence` column
All knowledge is a Fact. Inferred knowledge has lower confidence. No separate type.

**Advantages:** Simple. No schema change needed.

**Disadvantages:**
- No expiry semantics. A low-confidence Fact still lives forever.
- No source distinction. "Confidence 0.4" does not explain whether confidence is low because the user was uncertain or because it was derived from a small sample.
- Cannot filter "show me only verified knowledge" without an additional `is_verified` boolean that reintroduces the type distinction informally.
- Confidence decay has no trigger — when does a low-confidence Fact expire?

### Alternative B: Separate Observation entity (chosen)
Facts and Observations are distinct entities. Facts are verified and do not expire. Observations are inferred, carry confidence scores, derivation metadata, and an optional `expires_at`.

**Advantages:**
- Semantic clarity: Facts are reliable; Observations are probabilistic.
- Expiry is native to Observations, not a workaround.
- Source vs. derivation metadata are structurally different fields.
- The Prompt Engine can choose context composition strategy: "include all facts + high-confidence observations only."
- Confidence decay is implementable: a background job reduces observation confidence over time if not reinforced.

**Disadvantages:**
- Two entity types to manage.
- Some edge cases are ambiguous (is a calendar event a Fact or an Observation? — Fact, because it was explicitly stated).

### Alternative C: Separate Observation table + fact table with confidence
Keep Fact as-is but also add a confidence column to it. Add separate Observation table.

**Advantages:** More expressive for edge cases.

**Disadvantages:** Overengineering. If a Fact can have confidence, then the semantic boundary between Fact and Observation is blurred from the start.

---

## Decision

Adopt **Alternative B**.

```
Observation is not a degraded Fact. It is a different kind of knowledge claim.
```

A `Fact` asserts: "This is true, and I know how I know it."  
An `Observation` asserts: "This appears to be a pattern, with this confidence, until this date."

## Data Model

### Fact
```python
class Fact(Base):
    id: UUID
    workspace_id: UUID
    entity_id: UUID | None        # links to an Entity if applicable
    statement: str                # "Port 5432 is used by PostgreSQL"
    source_type: str              # "conversation" | "document" | "user_explicit"
    source_id: UUID               # conversation_id or document_id
    confidence: float = 1.0       # near 1.0; set by extractor
    created_at: datetime
    updated_at: datetime
    superseded_by_id: UUID | None # if this fact was corrected
```

### Observation
```python
class Observation(Base):
    id: UUID
    workspace_id: UUID
    entity_id: UUID | None
    statement: str                # "User tends to request summaries in the morning"
    derived_from: str             # "conversation_pattern" | "mission_statistics" | "rule_engine"
    derivation_agent: str         # which agent/engine produced this
    sample_size: int | None       # how many data points this was derived from
    confidence: float             # 0.0–1.0; can decay over time
    reinforcement_count: int      # incremented each time this pattern is re-confirmed
    created_at: datetime
    updated_at: datetime
    expires_at: datetime | None   # NULL means no expiry
    expired: bool = False         # set by background decay job
```

---

## Confidence Decay Rule

Observations that have not been reinforced decay according to:

```
new_confidence = confidence * decay_factor ^ (days_since_last_reinforcement / half_life_days)
```

Defaults: `decay_factor = 0.5`, `half_life_days = 30`. An Observation with initial confidence 0.8 that is not reinforced for 60 days will have confidence ≈ 0.2. When confidence drops below `0.15`, the Observation is marked `expired = True`.

---

## Prompt Engine Integration

The Prompt Engine injects knowledge into the system prompt as:

```
## Fatos conhecidos (verificados)
- Port 5432: PostgreSQL
- Servidor: Samsung NP550XCJ, 16GB RAM

## Observações (padrões inferidos, pode não ser 100% preciso)
- [0.7 confiança] Usuário prefere resumos pela manhã
- [0.6 confiança] Containers tendem a reiniciar após 48h com alto uso de memória
```

Low-confidence observations (below `0.4`) are excluded from the prompt context.

---

## Consequences

**Positive:**
- The system can express uncertainty natively.
- Stale inferences do not accumulate indefinitely.
- The Prompt Engine can choose context composition strategy by entity type and confidence threshold.
- Observation quality can be measured: what is the average confidence of our observations? How many have expired?
- The path to Bayesian updates and source credibility weighting is clear.

**Negative:**
- Edge cases require judgment: "Was this stated by the user or inferred from behavior?" — extractor agents must make this distinction correctly.
- Schema requires a background decay job that runs periodically.
- Two entities to manage vs. one.
