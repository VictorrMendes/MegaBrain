# ADR-004 — Capability Registry: Semantic Capabilities over Method Catalogs

**Status:** Accepted  
**Date:** 2026-06-25  
**Authors:** Architecture team

---

## Context

PAIOS uses a `CapabilityRegistry` as the directory of everything the system can do. Plugins register their capabilities here. The Planner queries the registry when building an execution plan.

The original design of the registry listed tool names and their signatures. The evolved design requires capabilities to be described semantically — as things the system can accomplish — rather than as a list of callable methods.

---

## Problem

If the CapabilityRegistry exposes tool signatures (`docker.restart(container: str)`), the Planner must know the exact method name to use it. This creates several problems:

1. **Refactoring breaks plans.** If `docker.restart` is renamed to `docker.container.restart`, every plan that referenced the old name is broken, including stored historical plans and workflow templates.

2. **LLM prompting is brittle.** A Planner prompt that lists 50 tool signatures is hard for an LLM to reason about. A Planner prompt that lists 8 capabilities with descriptions is significantly easier.

3. **No discovery by intent.** "Who can manage containers?" cannot be answered by a method catalog. It can only be answered by a semantic capability description.

4. **No permissions or dependency information.** A method catalog tells you what can be called. A capability descriptor tells you what is required (permissions, context) and what it affects (events produced/consumed).

5. **Tight coupling between Planner and plugin internals.** The Planner should not need to know that `docker.stats` and `docker.logs` both come from the same plugin.

---

## Why This Generates Future Difficulty

As the plugin ecosystem grows, the number of tools will grow faster than the number of capabilities. A mature system might have 200 tools across 20 plugins, but only 40 logical capabilities. Exposing 200 tool signatures to the Planner context window is both wasteful and confusing.

More importantly: the Planner's job is to reason about *what the system can accomplish*, not to know *how* it is implemented. Capability descriptions are the right interface between the Planner (which is an intelligence component) and the Plugin system (which is an infrastructure component).

---

## Alternatives Considered

### Alternative A: Method catalog registry
Registry contains function names, signatures and docstrings.

**Advantages:** Simple. Direct mapping from plan step to function call.

**Disadvantages:** Brittle to refactoring. LLM prompt grows with tool count. No semantic reasoning possible. No permission or context information.

### Alternative B: Capability descriptors with tool lists (chosen)
Registry contains semantic capabilities. Each capability has a human-readable description, permission requirements, context requirements, events it consumes/produces, and a list of tools it exposes.

**Advantages:**
- Planner works with intent ("who can manage containers?").
- Tool names are an implementation detail, changeable without breaking the Planner.
- Permission validation is declarative.
- LLM prompt context is bounded and meaningful.
- Capability discovery by tags, description similarity, or event contract.

**Disadvantages:**
- More verbose registration.
- Two layers of naming (capability name + tool name).
- Plugin authors must understand the distinction.

### Alternative C: Capability descriptors without tool lists
Capabilities are purely semantic; tools are never exposed.

**Advantages:** Maximum decoupling.

**Disadvantages:** The Executor cannot resolve which function to call for a given step. The plan must reference the capability, not the tool, making step execution impossible without an additional resolution step that reintroduces the tool concept.

---

## Decision

Adopt **Alternative B**.

Capabilities are the primary interface for the Planner. Tools are the internal implementation that the Executor uses. A plan step references a tool name; the CapabilityRegistry resolves which capability owns that tool and which function to call.

The separation:
- **Planner** reasons about capabilities (by description, tags, events).
- **Plan steps** reference tools (by name).
- **Executor** resolves tools via `capability_registry.get_tool(tool_name)`.
- **PlanValidator** checks that all tools in the plan exist in the registry and that the workspace holds the required permissions.

## Capability Schema

```yaml
name: container_management
description: >
  Manage Docker containers: start, stop, restart, inspect, stream logs,
  read real-time CPU/memory metrics, and execute commands inside containers.
plugin: docker
permissions:
  - infrastructure.docker.read
  - infrastructure.docker.write
required_context:
  - docker_socket_path
tools:
  - name: docker.start
    description: Start a stopped container
    parameters: { container: string }
  - name: docker.stop
    description: Stop a running container
    parameters: { container: string, timeout: integer }
  - name: docker.logs
    description: Stream the last N lines of container logs
    parameters: { container: string, lines: integer, since: string }
  - name: docker.stats
    description: Get real-time CPU and memory metrics
    parameters: { container: string }
events_consumed:
  - scheduler.fired
events_produced:
  - infrastructure.container.started
  - infrastructure.container.stopped
  - infrastructure.container.unhealthy
dependencies:
  - docker daemon reachable at docker_socket_path
tags:
  - infrastructure
  - docker
  - observability
```

---

## Consequences

**Positive:**
- Planner prompt context is compact and meaningful.
- Tool names can be refactored without changing how the Planner reasons.
- Permission validation is declarative and checked before execution.
- New capabilities are discoverable by the Planner semantically.
- Events produced by capabilities can trigger downstream reactions.

**Negative:**
- Plugin authors must write richer registration code.
- Two-layer naming adds slight complexity for developers new to the system.
- `required_context` must be verified at plan validation time, adding a check before execution.

---

## Impact on Other Modules

| Module | Impact |
|---|---|
| `kernel/capabilities/registry.py` | `Capability` dataclass expanded with `permissions`, `required_context`, `events_consumed`, `events_produced`, `dependencies` |
| `engines/plan/llm.py` | Uses `to_planner_context()` which returns capability descriptors (not tool lists) |
| `engines/plan/provider.py` | PlanValidator receives the full capability descriptor for permission check |
| `kernel/plugins/` | All plugins must be updated to register full capability descriptors |
| `main.py` | Plugin registration at startup populates CapabilityRegistry |
