# ADR-010 — Mobile-First Redesign & Progressive Web App

**Status:** Accepted  
**Date:** 2026-06-27  
**Phase:** 9.5 — Mobile Experience & PWA

---

## Context

Khonshu phases 1–9 built a complete cognitive OS — but the entire
interface was desktop-only. Zero responsive classes existed across
9,597 lines of component code. The shell used fixed widths throughout:
48px icon sidebar, 208px chat sidebar, 220px search bar, 288px side
panels. All interactive elements used `onMouseEnter`/`onMouseLeave`
hover patterns, and the minimum touch target was 24px — less than
half of Apple's 44px requirement.

The PWA baseline was equally absent: no `viewport` meta tag (mobile
browsers would render a zoomed-out desktop view), no manifest file,
no service worker, no icons, no safe-area support.

---

## Decision

### Navigation Strategy

**Desktop:** Existing `AppNav` icon sidebar (48px, vertical) remains
unchanged — it is hidden on mobile via `hidden md:flex`.

**Mobile:** New `MobileNav` bottom-bar component, shown only on mobile
via `flex md:hidden`. Displays 5 primary destinations (Dashboard, Chat,
Missions, Memory, Inbox) plus a sheet for the remaining 5. This matches
the iOS/Android native app navigation pattern users already know.

### Chat Layout Strategy

ChatPage currently uses a 3-column flex layout. On mobile this becomes
a **drawer pattern**:
- Main content (messages + input) is always full-width
- Conversation sidebar slides in from the left as a fixed overlay on mobile
- ReasoningPanel / ContextPanel appear as bottom sheets on mobile

This preserves the desktop layout completely (no regression) while
making mobile usable.

### PWA Strategy: Zero New Packages

Next.js 15 App Router natively supports PWA manifest generation via
`src/app/manifest.ts`, favicon via `src/app/icon.tsx`, and Apple
touch icon via `src/app/apple-icon.tsx` — all using the `ImageResponse`
API from `next/og`.

A custom service worker in `public/sw.js` handles offline caching using
the Fetch API natively (no Workbox dependency). Strategy:
- **App shell (HTML pages):** Network-first with cache fallback
- **Static assets (_next/static/):** Cache-first, permanent
- **API calls (/api/, /workspaces/):** Network-only (never stale data)

This avoids adding `@ducanh2912/next-pwa` or `next-pwa` as build-time
dependencies — keeping the `output: "standalone"` build clean.

### Responsive Approach

Mobile-first: all breakpoint modifiers use `md:` as the desktop
escalation point (768px). The existing Tailwind defaults are sufficient.
No new breakpoints were added.

Safe area insets (`env(safe-area-inset-*)`) are applied via CSS custom
properties and utility classes in `globals.css`. The MobileNav, TopBar
and content area all respect device notch and home indicator regions.

---

## Architecture Principles Applied

**Kernel unchanged.** This phase is frontend-only. No new engines,
no new models, no schema changes.

**No new abstractions.** `MobileNav` reuses the same `NAV_ITEMS`
constant from `AppNav`. No new routing logic — Next.js App Router
handles all navigation.

**Progressive enhancement.** Desktop users see zero change. The service
worker and PWA features activate only when installed as a PWA or on
capable browsers.

**Substitutability preserved.** The service worker caches only static
assets — API responses are always fetched live. The backend remains
completely unaware of the PWA layer.

---

## Alternatives Considered

### `@ducanh2912/next-pwa` or `next-pwa`
Rejected. These packages require build-time Workbox configuration,
add complexity to the standalone output, and are not necessary given
Next.js 15's native manifest route and simple caching needs.

### React Native / Expo
Out of scope for phase 9.5. Phase 17 (Mobile Companion) addresses
the native app.

### Separate mobile layout files
Rejected. CSS breakpoints are sufficient and keep the codebase DRY.
Separate layout files would require maintaining feature parity across
two codepaths.

---

## Consequences

**Positive:**
- Khonshu becomes usable on smartphones without a native app
- PWA installation gives home screen icon + standalone mode
- Partial offline support for recent sessions
- Zero new npm dependencies
- Desktop experience is unchanged

**Negative:**
- iOS PWA has limitations: no push notifications (requires native app),
  limited service worker scope
- Safe area insets require testing on physical devices (notch/island)
- The chat drawer pattern adds a state variable (`sidebarOpen`)
