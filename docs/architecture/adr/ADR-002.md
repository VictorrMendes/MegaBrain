# ADR-002 — Separate Domain Events from Infrastructure Events

**Status:** Accepted  
**Date:** 2026-06-25  
**Authors:** Architecture team

---

## Context

PAIOS uses an event bus as its primary coordination mechanism. Events flow between engines, agents, the scheduler and external notifications. The `EventType` registry currently lists both business-meaningful events (`mission.created`) and infrastructure state events (`plugin.loaded`).

As the system grows, more components will publish and subscribe to events. Without a clear boundary, domain engines will begin reacting to infrastructure signals, and infrastructure monitors will begin reacting to domain signals.

---

## Problem

Consider a future scenario: the Mission Engine subscribes to `container.started` (published by the Docker plugin when a container starts) to trigger an automatic health check mission. This subscription creates a direct dependency between the Mission Engine and the Docker plugin's event schema. When Docker is replaced with Podman, the event type changes, and the Mission Engine must be updated even though it has nothing to do with container technology.

This is the core problem: **coupling between domain logic and infrastructure state through the event bus**.

More concretely:
1. Domain engines that subscribe to infrastructure events become coupled to specific infrastructure choices.
2. Infrastructure monitors that subscribe to domain events become coupled to business workflows.
3. The `EventType` registry becomes unreadable: 50 events mixed with no clear boundary.
4. Testing domain components requires mocking infrastructure event sequences.

---

## Why This Generates Future Difficulty

PAIOS is explicitly designed to survive technology replacement. The LLM, database, event bus, and all plugins are replaceable. If domain engines depend on infrastructure event names, each infrastructure replacement cascades into domain changes. This violates the core principle stated in `ARCHITECTURE.md §0`.

As the capability and plugin ecosystem grows (Docker, HA, Notion, Calendar, Telegram, email, calendar, CI/CD), the surface of infrastructure events will grow faster than domain events. Without the boundary, domain components will progressively accumulate infrastructure subscriptions that are not their concern.

---

## Alternatives Considered

### Alternative A: No separation — all events in one registry
All events share the same registry and the same channel.

**Advantages:** Simple. No mental overhead.

**Disadvantages:** As described above, leads to coupling. Already happening with the current mixed `EventType` registry.

### Alternative B: Namespace separation — "domain.*" vs "infra.*"
Events are prefixed to indicate type. Domain engines only subscribe to `domain.*` patterns.

**Advantages:** Simple. No schema changes. Enforceable by naming convention.

**Disadvantages:** Convention-based. Nothing prevents a domain engine from subscribing to `infra.*`. Requires discipline, not architecture.

### Alternative C: Separate registries and bus channels (chosen)
Two distinct categories with separate documentation sections, and a convention enforced by code review and documentation:
- Domain events flow on `khonshu.events` (existing typed channel).
- Infrastructure events flow on `khonshu.infra` (new channel).
- Domain engines may only subscribe to the domain channel.
- Infrastructure monitors may subscribe to both.

**Advantages:**
- Physical separation makes violations visible.
- Domain components tested without any infrastructure event dependency.
- Infrastructure monitoring can evolve independently.
- New engineers learn the boundary from the channel name.

**Disadvantages:**
- Two channels to manage.
- Slightly more complex EventBus configuration.

---

## Decision

Adopt **Alternative C**. The domain/infrastructure separation is enforced by channel name and documented in `ARCHITECTURE.md §4`.

**Domain Events** (`khonshu.events`): facts about the business domain. Past tense. Meaningful to a non-technical stakeholder.  
**Infrastructure Events** (`khonshu.infra`): technical state changes. Present tense or past tense. Meaningful to a system administrator.

Domain engines are prohibited from subscribing to `khonshu.infra`. This prohibition is documented and enforced via code review. Future tooling may add a linting rule.

---

## Consequences

**Positive:**
- Domain engines are fully isolated from infrastructure state.
- Infrastructure can be replaced without touching domain logic.
- The `EventType` registry becomes readable: two clear sections.
- Domain component tests are simpler (no infrastructure event mocks).

**Negative:**
- Two bus channels instead of one.
- Any scenario where a domain engine legitimately needs to react to infrastructure state must be mediated by a bridge component (e.g., a Kernel-level adapter that translates `container.unhealthy` into a domain event `workspace.infrastructure_degraded`). This is the correct design but adds a small amount of indirection.

---

## Impact on Other Modules

| Module | Impact |
|---|---|
| `kernel/events/schema.py` | `EventType` class split into `DomainEventType` and `InfraEventType` |
| `kernel/events/bus.py` | `subscribe_event()` defaults to domain channel; `subscribe_infra()` for infrastructure |
| `kernel/agents/` | Existing workers already use the legacy dict channel; no change |
| All plugins | Infrastructure events go to `khonshu.infra`, not `khonshu.events` |
| Observability stack | Subscribe to both channels for full visibility |
