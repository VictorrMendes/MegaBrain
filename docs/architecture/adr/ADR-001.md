# ADR-001 — Separate Mission, ExecutionPlan and MissionStep as distinct entities

**Status:** Accepted  
**Date:** 2026-06-25  
**Authors:** Architecture team  
**Supersedes:** Original design that stored plan as JSONB on Mission

---

## Context

The initial implementation of the Mission system linked `MissionStep` records directly to a `Mission` record. The `plan()` method on `MissionEngine` generated steps and attached them to the mission. No intermediate entity existed between the Mission and its steps.

The consequence was that Mission and its execution strategy were the same object. When a mission needed replanning (after a plan failure or human rejection), the only option was to delete the existing steps and create new ones, losing the history of the failed plan.

---

## Problem

Treating Mission and ExecutionPlan as one entity creates several specific problems:

1. **Lost history on replan.** When a human rejects a plan or the planner fails, there is no record of what the first plan looked like. Debugging "why did this mission fail during planning?" becomes impossible.

2. **Approval granularity.** A human approving a mission is actually approving a plan — a specific strategy. If the plan is replaced, the approval must be re-requested. Without a separate `ExecutionPlan` entity, there is no object to which the approval is attached.

3. **No versioning.** Two plans for the same mission cannot be compared. "This plan used 8 steps; the revised plan uses 5. Here is why." becomes impossible to express.

4. **Metrics pollution.** If steps are deleted and recreated on replan, any metrics query (e.g., "average steps per mission") returns misleading results.

5. **Future parallel plans.** A/B testing two planners on the same mission (run both, apply the one that finishes first) is architecturally impossible when steps live directly on the mission.

---

## Why this generates future difficulty

As PAIOS evolves, the Planner will be one of the most iterated components. New LLMs, new prompting strategies, human-in-the-loop workflows, multi-planner ensembles — all require the ability to produce multiple candidate plans for the same mission. Building the system now without the `ExecutionPlan` intermediate entity means that adding this capability later requires a schema migration, data migration, and refactoring of all code that references `MissionStep.mission_id` directly.

The cost of introducing `ExecutionPlan` now is a single additional table and a foreign key change. The cost of introducing it later, when dozens of queries and services reference `MissionStep.mission_id`, is much higher.

---

## Alternatives Considered

### Alternative A: JSONB plan storage on Mission
Store the plan as a JSON array in a `plan JSONB` column on the `missions` table.

**Advantages:** Simple. No extra join. Plan visible without a second query.

**Disadvantages:**
- Impossible to query individual steps (e.g., "which tool failed most?").
- Impossible to version or compare plans.
- Plan cannot be updated atomically at the step level.
- No audit trail for step-level state changes.
- Explicitly rejected in the project's architectural discussions.

### Alternative B: MissionStep → Mission directly (current implementation)
Steps reference `mission_id` directly with no intermediate plan entity.

**Advantages:** Simpler schema. Already implemented.

**Disadvantages:**
- No replanning support without step deletion.
- No plan versioning.
- Approval attached to Mission, not to a specific plan.
- Acceptable as a transitional state but not as the final model.

### Alternative C: Separate ExecutionPlan entity (chosen)
A separate `execution_plans` table sits between `missions` and `mission_steps`.

**Advantages:**
- Full plan history preserved across replanning.
- Approval attached to the specific plan (version 1, version 2).
- Steps can be queried per plan, per mission, or globally.
- A/B planner comparison is architecturally possible.
- `ExecutionPlan.status` tracks its own lifecycle independently.

**Disadvantages:**
- Additional join required in most step queries.
- Schema migration required to move from current state.

---

## Decision

Adopt **Alternative C**. The `ExecutionPlan` table will be introduced in migration 007, changing `mission_steps.mission_id` to `mission_steps.execution_plan_id` (with `mission_id` retained as a denormalized convenience column).

The current implementation (Phase 2A) is treated as transitional. Code that reads `MissionStep.mission_id` will continue to work through the denormalized column. The `ExecutionPlan` entity will be the authoritative parent.

---

## Consequences

**Positive:**
- Replanning is a first-class operation: create new ExecutionPlan, mark old one `superseded`.
- Human approval is attached to a versioned, immutable plan.
- Full audit history of every plan attempted for a mission.
- Step-level metrics are accurate across replans.

**Negative:**
- All queries that join steps must go through `execution_plans`.
- Migration 007 will require a data migration (create one ExecutionPlan per existing Mission, update FK).
- Slightly more complex MissionEngine code for plan creation.

---

## Impact on Other Modules

| Module | Impact |
|---|---|
| `MissionEngine` | `plan()` must create an `ExecutionPlan` record before creating steps |
| `PlanProvider` | Return type changes from `list[MissionStep]` to `ExecutionPlan` |
| `PlanValidator` | Validates `ExecutionPlan`, not a step list |
| `routers/missions.py` | `GET /{id}` response includes current and historical plans |
| `schemas/mission.py` | New `ExecutionPlanResponse` schema |
| Migration 007 | New table, FK change, data migration |
